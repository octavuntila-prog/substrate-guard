# v13.2.16 (2026-04-22)

**Packaging hygiene release.** Addresses four latent defects discovered during
production upgrade and post-release empirical validation of v13.2.13/14/15.

## Fixes

### fix(scripts): persist executable bit in git tree

All shell scripts in `scripts/` were stored with mode `100644` (non-executable)
in the git index. On Linux production hosts, `git reset --hard` or fresh clones
would strip the exec bit, causing `cron` to fail with `Permission denied` when
attempting to invoke the scripts.

**Fix:** `git update-index --chmod=+x` applied to `audit.sh`, `cron-audit.sh`,
`deploy.sh`, `run-ci-local.sh`, `setup-cron.sh`, and `stack_audit.sh`. All
shell scripts now ship with mode `100755` in the git index.

**Why this matters:** On 2026-04-20, a silent cron failure occurred after a
`git reset --hard v13.2.14` on production — the scheduled 04:00 UTC audit
never ran. Root cause: the running deployment had `chmod +x` applied manually
on the host, but the git index had always tracked the scripts as 644. Reset
stripped the exec bit, and without `MAILTO` configured (and no MTA installed),
the failure produced no alert. This release prevents recurrence.

### fix(scripts/cron-audit.sh): TELEGRAM_CHAT_ID fallback to TELEGRAM_ADMIN_ID

The Telegram alerting block in `scripts/cron-audit.sh` required
`TELEGRAM_CHAT_ID` to be present in `.env`. On the production host, the
environment file contains `TELEGRAM_ADMIN_ID` (used by the `ai-research-agency`
backend for admin routing), not `TELEGRAM_CHAT_ID`. The alerting block silently
evaluated false on every run; no Telegram alert would have been sent even if
violations had been detected.

**Fix:** introduced `TELEGRAM_TARGET` intermediate variable with shell
parameter expansion fallback chain:
`TELEGRAM_TARGET="${TELEGRAM_CHAT_ID:-${TELEGRAM_ADMIN_ID:-}}"`.
The `grep` clause in line 51 was extended to extract both variables.
Environments with either variable present now produce working alerts.

### feat(audit): add substrate_guard_version field to audit JSON

Audit reports in `/var/log/substrate-guard/` previously had no reference to
the substrate-guard version that generated them. Cross-referencing metrics
from a report against the corresponding source code required guessing the
commit SHA from the timestamp and git log.

**Fix:** `substrate_guard_version` field added to audit JSON top-level,
immediately after `timestamp`. Example output:
```json
{
  "timestamp": "2026-04-22T10:30:00.123456",
  "substrate_guard_version": "13.2.16",
  "db_records": {...},
  ...
}
```

Reports generated by v13.2.16+ now self-document their provenance.
Reports generated by earlier versions (up to and including v13.2.15) do
not contain this field; the absence of the field is itself diagnostic
evidence of a pre-13.2.16 audit.

### chore(build): introduce scripts/release.sh mechanical release pipeline

Releases prior to v13.2.16 were cut manually: `git tag -a vX.Y.Z` plus
`git push origin vX.Y.Z` plus optional `gh release create`. Four mistakes
in manual release discipline were identified across v13.2.13–15:

- v13.2.13: `__version__` in `__init__.py` not synced with `pyproject.toml`
- v13.2.14: `tree-sitter` + `tree-sitter-bash` packaged incorrectly as dev
  dependencies; fresh `pip install substrate-guard` produced a silently
  broken installation
- v13.2.15: cryptography constraint too restrictive vs. actual API usage
- v13.2.16: four distinct fixes needed, easy to forget or mis-scope

**Fix:** `scripts/release.sh` now enforces a mechanical pipeline:
1. Validate semver format of tag argument
2. Check working tree clean + on `main` branch
3. Check tag does not already exist (local or origin)
4. Verify `pyproject.toml` version matches tag
5. Verify `__init__.py` `__version__` matches tag
6. Verify `docs/releases/vX.Y.Z.md` exists and non-empty
7. Run pytest (must pass, or explicit user override)
8. Create annotated tag with release notes as tag message
9. Push branch + tag
10. Optionally create GitHub Release via `gh` CLI

Manual `git tag` outside `release.sh` is now forbidden by convention (this
is Discipline 3 from SUBSTRATE_GUARD_ROADMAP). v13.2.16 is the first release
cut through this pipeline; the script validates itself by producing this
release.

### chore(build): add .gitattributes for line ending enforcement

Windows environments (`C:\Users\*\substrate-guard\`) may introduce CRLF line
endings into shell scripts during editing. On Linux execution, CRLF-terminated
shebangs produce obscure errors (`bad interpreter: No such file or directory`
with no visible cause).

**Fix:** `.gitattributes` enforces LF line endings for `*.sh` and `*.py`:
```
*.sh text eol=lf
*.py text eol=lf
```

## Non-changes

No changes to `substrate_guard.ast_parse`, `substrate_guard.policy.engine`,
`substrate_guard.chain`, `substrate_guard.observe`, `substrate_guard.attest`,
`substrate_guard.offline`, or `substrate_guard.verify`. No changes to audit
semantics, policy rules, or chain integrity algorithms. This is purely a
packaging and operational hygiene release.

AST benchmark corpus metrics are unchanged: 5 detection divergences, 48.7%
regex detection, 82.1% AST intended detection, 69.2% AST observed detection,
12.9-point gap. No re-run of the corpus required.

## Upgrade path

From v13.2.15 to v13.2.16: no breaking changes. Safe to pull and restart
cron. Audit report schema adds one new top-level field; consumers parsing
reports should handle the additional field gracefully (no field removal,
no type changes).

This release closes the operational hygiene gap identified in the 2026-04-20
