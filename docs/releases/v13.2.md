# substrate-guard v13.2 — ProcessEvent ↔ CLI verifier (optional)

Formal CLI verification (`verify_artifact` with `artifact_type="cli"`) can run on **observed process events** when enabled: the command line is reconstructed from `ProcessEvent.args` (or `filename`) and passed through the same regex-based `verify_cli` path as manual CLI checks. Default remains **off** on `Guard` for backward compatibility; **Black Box** commands `demo`, `export`, and `stack-benchmark` enable it by default (opt out with `--no-verify-process-cli`). **`substrate-guard monitor`** supports **`--verify-process-cli`** and **`--no-verify-process-cli`**. **`SubstrateGuard`** accepts **`verify_process_cli=`**, JSON **`verify_process_cli`**, and environment **`SUBSTRATE_GUARD_VERIFY_PROCESS_CLI`** (`1`/`on`/`0`/`off`).

## What changed

| | v13.1 | v13.2 |
|--|-------|-------|
| Tests | 328 | **348** |
| `Guard` | `verify_artifact` only | **`verify_process_cli`** optional flag |
| `ProcessEvent` in pipeline | policy + optional chain | **+ same CLI verifier as manual `verify --type cli`** when flag on |
| CLI demos | full stack without per-exec CLI | **`demo` / `export` / `stack-benchmark`** default on; **`SUBSTRATE_GUARD_VERIFY_PROCESS_CLI=off`** disables without flags |
| Observability | formal count only | **`SessionReport.cli_process_verifications`**, **`layers.verify.cli_process_checks`** in `to_dict()`, **`summary_line`** suffix `cli_exec_checks=` |

## Behavior

- **`Guard(..., verify_process_cli=True)`** (requires `verify=True` and Z3 available): for each `ProcessEvent`, after the observe chain entry (if `chain` is enabled), `evaluate_event` calls `verify_artifact(cmd, artifact_type="cli", ...)` so `GuardEvent.verification` and chain **`formal_verification`** entries match the manual CLI path.
- **Precedence (SubstrateGuard):** explicit constructor argument **>** **`SUBSTRATE_GUARD_VERIFY_PROCESS_CLI`** **>** JSON **`verify_process_cli`**.
- **Pipeline commands:** `DEFAULT_VERIFY_PROCESS_CLI_PIPELINE` in `combo_cli` (currently `True`); `--no-verify-process-cli` or env **`0`/`false`/`off`** turns off the default for CI or latency runs.
- **Performance:** enabling this on high-frequency exec paths increases work per event; use **`--no-verify-process-cli`** on benchmarks or demos when measuring policy-only latency.

## Compatibility

Backward compatible: default **`verify_process_cli=False`** on `Guard`. Existing callers are unchanged unless they opt in. New **`SessionReport.cli_process_verifications`** defaults to **0**.
