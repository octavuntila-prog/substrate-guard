# v13.3.1 (2026-04-30) — datetime.utcnow() Deprecation Fix

substrate-guard v13.3.1 is a patch release that migrates four callsites
of the deprecated `datetime.utcnow()` to `datetime.now(timezone.utc)`
via a new helper `_utcnow_naive()`. The migration is intentionally
scoped to silencing the deprecation warning while preserving the
v13.3.0 JSON schema contract byte-for-byte.

## Deprecation Fix

`datetime.utcnow()` is deprecated in Python 3.12 (production server
runs 3.12.3) and scheduled for removal in a future Python release.
Production cron emitted the corresponding `DeprecationWarning` on
every run since the M1.2 deploy.

Four callsites migrated, all in `substrate_guard/audit.py`:

| Line | Context |
|---|---|
| 262 | DB fetch window for `pipeline_traces` |
| 274 | DB fetch window for `agent_runs` |
| 445 | `"timestamp"` field in audit report JSON |
| 490 | Filename component for `audit_YYYYMMDD_HHMMSS.json` |

All four now call `_utcnow_naive()`, defined in the same module:

```python
def _utcnow_naive() -> datetime:
    """UTC now as naive datetime.

    Replacement for ``datetime.utcnow()``, which is deprecated in
    Python 3.12+ and scheduled for removal. Returns a naive datetime
    (no tzinfo) to preserve exact JSON schema compatibility with
    v13.2.x and v13.3.0 audit reports — ``isoformat()`` output does
    NOT include the ``+00:00`` suffix.

    For new code that needs timezone-aware datetimes, use
    ``datetime.now(timezone.utc)`` directly instead of this helper.
    """
    return datetime.now(timezone.utc).replace(tzinfo=None)
```

## Schema Preservation

Audit report JSON schema is **byte-identical** to v13.3.0. The
`"timestamp"` field retains naive ISO 8601 format with no `+00:00`
suffix. Verified empirically against production output
`audit_20260430_040002.json`:

```
v13.3.0 server: "timestamp": "2026-04-30T04:00:02.756369"
v13.3.1 helper: "timestamp": "2026-04-30T06:09:34.749835"
                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ format pattern identical
```

DB query semantics unchanged — `since` filters at lines 262/274
continue to produce naive datetimes for compatibility with existing
PostgreSQL column types.

## Non-changes

- **Audit JSON schema unchanged.** Still 13 keys (11 baseline + 2
  M1.2 metadata). No new fields, no removed fields, no format
  changes.
- **Guard class API unchanged.** Public surface area identical to
  v13.3.0.
- **Cron behavior unchanged.** `cron-audit.sh` continues to invoke
  `python -m substrate_guard.audit` without modification.
- **Default policy behavior preserved.** `policy_engine: builtin`,
  `policy_engine_source: default` per v13.3.0 contract.

## Forward Path

A future minor (v13.4.0 or later) may modernize to fully
timezone-aware datetimes. Such a release would explicitly notice the
schema delta (`"timestamp": "...+00:00"` instead of naive). v13.3.1
is intentionally scoped to silencing the deprecation warning only —
no schema modernization in this release.

## Upgrade path

### From v13.3.0 (no behavior change)

```bash
cd /opt/substrate-guard
git fetch origin --tags
git checkout v13.3.1

python -c "import substrate_guard; print(substrate_guard.__version__)"
# Expected: 13.3.1
```

Cron continues working identically. Next scheduled run produces
v13.3.1 output with no DeprecationWarning emitted to
`cron_YYYYMMDD.log`.

```bash

```

## Thanks

Issue surfaced 2026-04-30 during day 6 of the M1.2 baseline
observation window — the cron log tail revealed the deprecation
warning, which had been latent in v13.3.0 and earlier. Investigation
completed in under 30 minutes: 4 callsites enumerated, schema
preservation path confirmed, helper designed, patch applied,
empirical verification against production output. Single-file fix,
single-commit release.

Design pattern: separate "silence warning" patch (this release) from
"modernize datetime semantics" minor (future). The helper makes the
M1.2-era workaround greppable and self-documenting for future
maintainers.
