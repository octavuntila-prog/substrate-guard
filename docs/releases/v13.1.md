# substrate-guard v13.1 — Verifier Integration & Audit Chain Closure

The verifier integration release. Five subsystems (**cli**, **tool**, **code**, **hw**, **distill**) now route correctly through `Guard.verify_artifact`, formal verification results land in the HMAC chain alongside policy decisions, and counterexamples are human-readable instead of `repr()` dumps.

## Bug fixes

- **`Guard.verify_artifact` (cli):** Was reading `result.get("verified")` from a `CLISafetyResult` that exposes **`safe`**. Fixed with explicit mapping via `_map_verification_to_guard`.
- **`Guard.verify_artifact` (tool):** Was calling `ToolVerifier().verify_tool()`, but `verify_tool` is a **module** function, not a method. Replaced with `tool_definition_from_payload(...)` + `verify_tool(...)`.
- **`Guard.verify_artifact` (code / hw):** Was passing raw `dict`s where `CodeVerifier.verify(spec)` and `HardwareVerifier.verify(hw_spec)` expect typed objects. Added **`spec_from_mapping`** and **`hw_spec_from_mapping`**.
- **Distill:** `DistillationVerifier` had no public alias matching the import name **`DistillVerifier`**. Added the alias and a `verify(artifact)` entry point for JSON payloads.

## Chain extension

Formal verification results are persisted in the HMAC chain as **`formal_verification`** entries next to observe-driven events and policy annotations. Each entry includes **`agent_id`**, **`verifier_type`**, **`verified`**, **`artifact_preview`**, **`counterexample`**, and **`proof_time_ms`**. Auditors can trace not only *that* something was blocked, but *why*, with the same tamper-evidence guarantees as the rest of the chain.

## Honest gap inventory

The test **`test_cli_honest_gap_inventory`** documents commands that the current **regex layer** still classifies as SAFE on purpose (frontier for the next rules or for non-regex approaches). Examples evolve with the pattern list; the intent is **disclosure**, not hidden gaps. Heavier cases (e.g. structural / AST analysis) are out of scope for pure regex.

## Compatibility

Fully **backward compatible**. No breaking changes to public entrypoints intended for external callers. Existing `GuardEvent` consumers keep working; new chain entries are **additive**.
