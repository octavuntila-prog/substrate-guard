# v13.3.0 (2026-04-24) — Configurable Policy Engine

substrate-guard now supports user-configurable policy engine selection
at runtime. The built-in Python rule engine remains the default; an
OPA/Rego engine becomes available via CLI flag or environment variable.
Audit JSON reports include two new top-level fields recording which
engine evaluated the events and how that choice was resolved.

This release is the first milestone of FAZA 1 from the M1.2 design
document. It addresses the semantic gap documented in the substrate-guard
paper Section 5.3.1: through v13.2.x, the built-in engine was active
by default due to a hardcoded policy path that did not resolve to any
filesystem location. v13.3.0 makes this choice explicit and overridable
while preserving the same default behavior.

## Features

### feat(audit): --policy CLI flag

New flag controls policy engine selection per-invocation:

```bash
python -m substrate_guard.audit --policy builtin    # Python built-in (default)
python -m substrate_guard.audit --policy rego       # OPA/Rego engine
```

The `--policy rego` mode requires the `opa` binary on PATH. See
`docs/rego-setup.md` for installation instructions.

### feat(audit): SUBSTRATE_GUARD_POLICY environment variable

Global activation without CLI changes:

```bash
export SUBSTRATE_GUARD_POLICY=rego
python -m substrate_guard.audit    # uses rego automatically
```

Resolution precedence: CLI flag > environment variable > hardcoded
default (`builtin`). Invalid environment values log a warning and fall
through to the default; invalid CLI values raise at startup.

### feat(audit): policy_engine and policy_engine_source in JSON reports

Audit JSON reports now include two top-level fields:

```json
{
  "policy_engine": "builtin",
  "policy_engine_source": "default"
}
```

`policy_engine_source` values: `cli` (flag passed), `env` (env var set),
`default` (neither). Enables reproducibility verification and debugging
("why did this run use rego?" → "the env var set it").

### feat(cli): parity across both entry points

The `substrate-guard audit` CLI wrapper (via `cli.py`) has identical
flag support to `python -m substrate_guard.audit`. Both resolve policy
mode the same way, emit the same startup log line, and write reports
with the same schema.

### feat(cron): opt-in documentation

`scripts/cron-audit.sh` includes a comment block documenting how to
activate Rego enforcement via environment variable in production cron.
No behavioral change to existing cron deployments — default remains
built-in.

## Non-changes

- **Default behavior preserved exactly.** Running `python -m substrate_guard.audit`
  with no flags and no environment variable produces output structurally
  identical to v13.2.16 output, with the two new metadata fields added.
  Existing JSON consumers that parse `d['evaluation']['violations']`
  continue to work.
- **Guard class API unchanged.** `Guard.__init__(policy=...)` signature
  is identical to v13.2.x. The `policy=` parameter still accepts a path
  string; internally, the new `BUILTIN_POLICY_PATH` sentinel triggers
  the Python built-in fallback the same way `"nonexistent/"` did.
- **Rego policy file unchanged.** `substrate_guard/policy/policies/agent_safety.rego`
  is shipped as in v13.2.x. No policy authoring changes in this release.
- **Cron-audit.sh behavior unchanged.** Existing production cron jobs
  continue to run the built-in engine without modification.

## Upgrade path

### From v13.2.16 (no behavior change)

```bash
# On production server:
cd /opt/substrate-guard
git fetch origin --tags
git checkout v13.3.0

# Verify version:
python -c "import substrate_guard; print(substrate_guard.__version__)"
# Expected: 13.3.0

# Cron continues working identically. Next scheduled run produces
# v13.3.0 output with the two new metadata fields.
```

### Activating Rego (separate decision)

See `docs/rego-setup.md` for OPA installation and Rego activation
workflow. Recommended to validate with baseline observation (7 days
on default) before switching engines.

### Rollback

```bash
# Return to v13.2.16:
cd /opt/substrate-guard
git checkout v13.2.16
# Cron next run uses v13.2.16 (no reinstall needed)
```
