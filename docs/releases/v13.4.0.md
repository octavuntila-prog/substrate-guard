# v13.4.0 (2026-05-09) — Chain Wiring (M0.3 Closure)

substrate-guard v13.4.0 wires the HMAC-SHA256 tamper-evident audit
chain into the production cron path. Pre-v13.4.0, `chain.py` existed
as implemented capability but was never invoked by `audit.py` —
`Guard()` was constructed without `chain=True`, so `self._chain` was
None throughout cron runs and no chain entries were produced.
v13.4.0 closes this gap. M0.3 milestone: the third Layer becomes
empirically active in production cron, not just in `combo_cli`
synthetic-scenario demos.

## Architectural decisions (locked)

Three decisions locked in `IMPLEMENTATION_PLAN_v13.4.0.md`,
preserved through Faza A:

1. **Random fallback opt-in strict.** `AuditChain()` raises
   `ChainConfigError` if no secret provided (parameter, env var, or
   explicit `allow_random_fallback=True`). Pre-v13.4.0 silently fell
   back to `os.urandom(32)` — chain worked but was unverifiable
   across runs. New behavior prevents silent loss of cross-run
   verifiability.
2. **Per-run chain (NOT master chain).** Each cron run exports
   `chain_YYYYMMDD_HHMMSS.jsonl`, self-contained. Cross-run
   prev_hash continuity deferred to v13.5+.
3. **HMAC key location: `/etc/substrate-guard/hmac.key`.** Standard
   sysadmin path, file mode `600` (or `400`), loaded by
   `cron-audit.sh` and exported as `SUBSTRATE_GUARD_HMAC_SECRET`.

## Changes

### feat(chain): Chain wiring in cron path (#37, core)

`audit.py` Guard instantiation now passes `chain=True` and
`hmac_secret`:

```python
hmac_secret = os.environ.get("SUBSTRATE_GUARD_HMAC_SECRET")
guard = Guard(observe=True, policy=policy_path, verify=True,
              use_mock=True, chain=True, hmac_secret=hmac_secret)
```

After audit JSON save, per-run chain export:

```python
if guard._chain is not None and guard._chain.length > 0:
    chain_path = f"{output_dir}/chain_{ts}.jsonl"
    guard._chain.export(chain_path)
```

A `Chain not exported (zero events)` log line distinguishes
"chain wired but no events" from "chain not wired" for M0.7
baseline observability.

### feat(audit): Compliance summary export (Addition 3, post-#37 unblock)

After chain export, generates compliance summary JSON via
`ComplianceExporter.export_summary()` written to
`compliance/summary_YYYYMMDD_HHMMSS.json`. Full SOC 2 / ISO 27001 /
ISO 42001 exports remain manual via `combo_cli` — they require
`SessionReport` context that production cron's batch-DB pipeline
does not produce.

### feat(ops): HMAC key management in cron-audit.sh

`scripts/cron-audit.sh` loads HMAC secret from
`/etc/substrate-guard/hmac.key`. Aborts with FATAL message if file
missing or permissions are insecure (must be 600 or 400). The
`cron-with-alert` wrapper sends a Telegram alert if
`cron-audit.sh` exits non-zero — fail-loud chain is operator-
visible without manual log inspection.

### fix(tests): Server pytest collection (#39 partial)

`pyproject.toml`:

```toml
pythonpath = [".", "tests"]   # was: ["tests"]
```

Adding `"."` makes pytest find `substrate_guard` from any CWD on
any environment without requiring `pip install -e .`. Server
pytest collection expected to match laptop's 405 post-deploy
(was 330 with errors). The `deploy.sh` portion of #39 was scoped
out — `deploy.sh` is not in current Gate 3 procedure.

### fix(combo_cli): BUILTIN_POLICY_PATH consistency (#41)

Five callsites in `combo_cli.py` (lines 170, 237, 272, 322, 385)
replaced literal `"nonexistent/"` placeholder with
`BUILTIN_POLICY_PATH` constant. Post-v13.3.2 sentinel-aware
dispatch matches the canonical sentinel literal, so legacy
`"nonexistent/"` produced WARNING (not INFO). Updated callsites
now produce consistent INFO logging.

## Migration / Operator setup

### Pre-deploy: HMAC key generation (one-time)

```bash
sudo mkdir -p /etc/substrate-guard
sudo openssl rand -hex 32 > /etc/substrate-guard/hmac.key
sudo chmod 600 /etc/substrate-guard/hmac.key
```

The key is 64 hex characters (32 bytes raw). Store offline backup
in your password manager — losing the key means losing chain
verifiability for past exports.

### Post-deploy: First cron run validation

After `git checkout v13.4.0`, the next cron run (04:00 UTC)
produces three files:

```
/var/log/substrate-guard/audit_*.json              (existing)
/var/log/substrate-guard/chain_*.jsonl             (NEW)
/var/log/substrate-guard/compliance/summary_*.json (NEW)
```

Verify chain integrity:

```bash
python3 -c "
from substrate_guard.chain import AuditChain
ok, reason = AuditChain.verify_export(
    '/var/log/substrate-guard/chain_YYYYMMDD_040002.jsonl',
    secret=open('/etc/substrate-guard/hmac.key').read().strip()
)
print('Chain integrity:', 'INTACT' if ok else 'BROKEN:', reason)
"
```

## Upgrade path

### From v13.3.2

```bash
# Pre-deploy: setup HMAC key (one-time)
sudo openssl rand -hex 32 > /etc/substrate-guard/hmac.key
sudo chmod 600 /etc/substrate-guard/hmac.key

# Deploy
cd /opt/substrate-guard
git fetch origin --tags
git checkout v13.4.0
python3 -c "import substrate_guard; print(substrate_guard.__version__)"
# Expected: 13.4.0
```

If the HMAC key is missing at first cron run, `cron-audit.sh`
exits 1 with FATAL message + Telegram alert via `cron-with-alert`.
Operator response: generate the key, verify permissions.

## Rollback

```bash
cd /opt/substrate-guard
git checkout v13.3.2
```

Cron returns to v13.3.2 behavior on next 04:00 UTC. The HMAC key
file at `/etc/substrate-guard/hmac.key` can remain in place
(unused on v13.3.2). Chain/compliance file production stops.

Abort criteria for rollback:
- First cron run on v13.4.0 fails (non-zero exit, no audit JSON)
- Chain export produces invalid JSONL (verify_export BROKEN)
- M0.7 baseline window day 1 datapoint missing or malformed
