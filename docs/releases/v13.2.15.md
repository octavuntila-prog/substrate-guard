# substrate-guard v13.2.15 — packaging: lower cryptography constraint

Single-fix release aligning the `cryptography` dependency spec with the
operational reality observed during the v13.2.14 production deployment.

## Changes

- **fix(deps)**: Lowered `cryptography>=46.0.6` → `cryptography>=3.4`.
  An audit of `substrate_guard/attest/device_key.py` — the only file in
  the package that imports `cryptography` — found exactly two API
  surfaces in use:

  - `hazmat.primitives.serialization` (stable since 0.6, 2014)
  - `hazmat.primitives.asymmetric.ed25519.Ed25519PrivateKey` (introduced
    in 2.6, February 2019)

  Both predate `cryptography 3.0` by a wide margin. The previous
  `>=46.0.6` bound was empirical overreach: the v13.2.14 production
  deployment at 89.167.66.225 ran cleanly against the stock Ubuntu 24
  `cryptography 41.0.7` — 96 audit events, 0 Layer 5 signing errors,
  5.22 ms/event — which on its own falsified the `>=46.0.6` claim.
  `>=3.4` is the oldest release series still receiving security
  backports in the wider Python ecosystem, so anchoring there gives a
  floor that is both defensibly secure and broadly compatible.

No API changes, no policy changes, no schema changes, no behavioural
changes. This release is pure dependency metadata. Detection output on
the AST benchmark corpus is bit-identical to v13.2.14.

## Impact

v13.2.15 unblocks `pip install substrate-guard` on, among others:

- Ubuntu 22.04 LTS (stock `cryptography` 3.4.8)
- Debian 12 bookworm (stock 38.0.4)
- RHEL 9 / Rocky 9 / AlmaLinux 9 (stock 36.0.1)
- Corporate Python environments where `cryptography` upgrades require
  separate security review

Prior to this change these systems would silently require a
`pip install --upgrade cryptography` before `substrate-guard` itself
could resolve — a real barrier for reviewers reproducing the Paper #5
numbers on locked-down systems.

## Note for paper #5 readers

The AST benchmark corpus measurements are unaffected: v13.2.15 changes
*which environments* can install substrate-guard cleanly, not what
substrate-guard detects. For strict version-pinned reproducibility:

    pip install substrate-guard==13.2.15

    python scripts/verify_live.py --version v13.2.15

Expected output: 47 samples evaluated, 0 errors, 5 detection
divergences, 0 rule-name divergences — identical to v13.2.14 and
v13.2.13.
